## 一、开场 60 秒介绍（建议背熟）

**你可以这样说：**

1. **主数据 + Translation 表**，内容和语言解耦  
2. **滚动锚点切章 + S 形路径**，体验接近参考 Demo

## 二、项目与方案类

### Q1：为什么选这个技术栈？

| 层级 | 选型 | 口述要点 |
|------|------|----------|
| 前端 | Expo RN + TypeScript | 岗位要求移动端；Web 预览方便二测录屏；一套代码多端 |
| 后端 | Express + Prisma | 与前端同语言，研发快；Prisma schema 即文档 |
| DB | SQLite | Demo 轻量；设计可迁 PostgreSQL（表结构不变） |
| i18n | i18next + Translation 表 | UI 壳与 CMS 内容分离，内容运营可独立扩语言 |

**加分句：**「如果上线用户多，后端换 PG + 连接池，前端加鉴权与离线缓存，架构不用推倒重来。」

### Q2：核心用户路径是什么？你如何拆功能？

**答：** 旅程浏览 → 节点弹窗 → 封面 → 分段阅读 → 完成评分 → 回地图解锁。  
计划书里按 Tab / 地图 / 章节 / 弹窗 / 学习流 / 解锁 / i18n / 长文八块写目标与验收，保证「先做主路径，占位 Tab 不挡验收」。

### Q3：4 天排期怎么排的？有没有偏差？

**答：** D1 后端 Schema/Seed/API → D2 地图 UI 联调 → D3 学习流与 i18n → D4 动效打磨、录屏、APK、文档。  
实际偏差在体验还原（路径、切章、阅读动画）比脚手架更费时间；优先保主流程可演示，再迭代视觉细节。

### Q4：Demo 刻意没做什么？为什么？

**答：** 今日/探索/陪伴是占位；无登录；单用户；SQLite。  
目的是把考核重点放在「能讲清方案 + 跑通主流程」。已知限制已在计划书第十章写出，并给了后续计划（JWT、PG、离线同步）。

## 三、后端 / 数据类（高频）

### Q5：为什么用主表 + Translation，而不是中英字段平铺？

**答：** 平铺扩展语言要改表；Translation 用 `(entityId, locale)` 唯一约束，加 `ja-JP` 只需插行。文案由内容团队改库即可，不必发版前端。

### Q6：讲一下解锁逻辑。

**答：**

1. Seed：第一章第 1 课 `available`，其余 `locked`  
2. `POST /lessons/:id/complete`：写 progress、星星 `+= starReward`  
3. 同章 `sortOrder + 1` → `available`；若已是章末 → 下一章第 1 课 `available`  
4. 旅程接口聚合 chapters + progress，返回前端使用的 `status`

### Q7：`GET /journey` 返回了什么？`sections` 干什么？

**答：** 两章课程、用户星星、`activeChapterId`、每课 status。  
`sections` 是路径上的分隔条：当前只有第二章用 `beforeLessonOrder: 1`，在第一章之后、第二章首节点前画「思维」分隔线——**不是课程实体**，是展示元数据。

### Q8：语言从哪里解析？默认为什么是中文？

**答：** 优先级：`?locale=` > `Accept-Language` > 默认 `zh-CN`。  
国内演示默认中文；Profile 切英文后前端同步写 locale 并 `cache: 'no-store'` 重新拉接口，避免缓存导致中英混排。

### Q9：SQLite 够不够？生产怎么演进？

**答：** Demo 够用。生产迁 PostgreSQL，Prisma 改 datasource；加连接池、索引（`userId+lessonId`、`locale`）、读写分离按需。长文可考虑对象存储 + CDN，库只存元数据与正文引用。

### Q10：没有鉴权怎么办？

**答：** Demo 写死 `userId=1`。上线会加登录（JWT/OAuth），所有 progress/profile 挂在真实用户上；敏感接口加中间件，仅完成自己的课。

## 四、前端 / 交互类（结合本 Demo 高概率深挖）

### Q11：旅程地图节点怎么排布？为什么不用左/中/右三档？

**答：** 早期三档太「硬」。现在 `pathLayout.ts` 用**像素级 `translateX` 序列**画 S 曲线，并按屏宽相对 390 缩放，更接近 Duolingo / 参考视频的蜿蜒感。

### Q12：滚动时章节头怎么切换？踩过什么坑？

**答：** 扁平 ScrollView；每章在首节点前放 **1px 锚点**，`onLayout` 记 Y；滚动时用 `scrollY + 96` 与锚点比较更新 `activeChapterId`。  

**坑：**

- 锚点挂在嵌套子 View 上时 `layout.y` 是相对父级，几乎为 0 → 切章失灵  
- 用「视口比例」当阈值时大屏提前切章 → 改回固定 96px  
- 标题切得晚会出现「看起来第 1 章 11 个点、第 2 章 3 个点」——其实 14 点都在，是切章锚点偏晚

### Q13：当前节点的星星效果怎么做的？

**答：** 不是绕节点公转，而是右上角**一小簇大小不一的星星**（四角星 + 小圆点）自转 + 透明度/缩放脉冲，贴合参考图。

### Q14：长文为什么要分段 Continue，而不是一次滚完？

**答：** 对齐参考产品「Continue 加载下一块」的节奏感；进度条按 `revealedCount / sections.length` 更可控。  
实现：`splitContentPages` 切 HTML → Continue 进入 loading（**菊花在正文底部**）→ `RevealContent` 高度动画自上而下展开。

### Q15：i18n 前端怎么保证和后端一致？

**答：**

- UI：`i18next` 管 Tab、按钮  
- 内容：journey/lesson API 带 `locale`  
- 切换语言：更新 profile + 清空旅程状态 + `key={locale}` 强制刷新 Tab  
- 请求 `cache: 'no-store'` 防浏览器缓存旧语言 JSON

### Q16：为什么 Sheet 不负责切章？

**答：** 产品期望是「滑地图自动换章」；Sheet 只展示**当前滚动章节**详情，避免两套状态互相打架。

### Q17：Web 和 APK 差异？

**答：** Web 录屏方便；原生动效更跟手。Android 默认 API `10.0.2.2` 指本机后端；真机要写局域网 `extra.apiUrl` 并开 cleartext（已开）。

## 五、工程与交付类

### Q18：如何复现 Demo？

**答：**

```bash
# API
cd spark-demo-api && npm i && npm run db:reset && npm run dev

# App
cd spark-demo-app && npm i && npm run web
```

Seed 后应是 **两章各 7 课 / 共 14 节点**；启动日志会打印课时数。  
APK：`docs/spark-demo.apk`；录屏：`docs/录屏文件.mp4`。

### Q19：APK 为什么是本地打而不是 EAS？

**答：** 本机未登录 Expo 账号时云端 build 不可用；用 `expo prebuild` + `gradlew assembleRelease` 产出可安装包。已配置 `eas.json` preview→apk，登录后可切云端构建。交付物已进仓库。

### Q20：Git 大文件怎么处理？

**答：** 录屏约 21MB、APK 约 68MB，已进仓；GitHub 对 >50MB 有警告但未超 100MB 硬限。后续生产制品更适合 Git LFS 或对象存储外链，仓库只留文档链接。

## 六、AI 协作类（本岗位极可能问）

### Q21：你怎么用 AI？哪些不能交给 AI？

| AI 适合 | 必须人工 |
|---------|----------|
| Schema / 路由样板 | 验收标准、功能边界 |
| 长文与中英翻译初稿 | 对照视频调间距、切章、动效 timing |
| 组件脚手架 | 发现并修「锚点 Y=0」「中英混排」等真机问题 |
| 文档润色 | 拍板取舍（例如去掉第一章中段分隔线） |

**话术：**「AI 加快 40%–60% 的样板代码，但体验验收和边界条件靠人对着参考一帧帧对齐。」

### Q22：如果 AI 生成了错误方案怎么办？

**答举真例：** 「继续探索」分隔线一度误导人以为第一章只有 4 点 → 删掉第一章分隔、把「思维」放到第二章起点；切章算法也曾用错坐标系。流程是：**跑一遍 → 对参考图 → 修数据或布局 → 更新计划书迭代对照**。

### Q23：AI App 岗位和「会写 CRUD」的区别？

**答：** 不止调模型 API。本 Demo 虽未接大模型，但结构可扩展：陪伴 Tab 可接对话；听记/摘要可挂异步任务。全栈要保证内容、进度、多语言、端到端体验连成产品，而不是孤立接口。

## 七、系统设计追问（高级岗常扩）

### Q24：如何支持十万用户同时学？

要点：**PG + 连接池**；progress 热点行考虑乐观锁/按用户分片；静态资源与长文 CDN；旅程接口缓存「章节结构」与「用户进度」分层；完成课用短事务 + 幂等（同一课重复 complete 不重复加星）。

### Q25：离线怎么做？

要点：已学过的正文缓存本地；进度队列离线完成，上线后同步冲突策略（以服务端为准或合并）；地图可先渲染缓存 skeleton。

### Q26：内容运营如何上新课？

要点：后台写 Chapter/Lesson + Translation；Seed/CMS 都可；前端旅程无改代码，最多刷新。分隔线、`sections` 继续当展示配置。

### Q27：监控与质量？

要点：API 结构化日志 + 错误率；关键路径（complete、unlock）埋点；E2E 测「完成第 1 课 → 第 2 课 available」；截图/录屏回归切章与 i18n。

## 八、行为与协作

### Q28：和产品目标不一致时怎么办？

**答结合本项目：** 产品要滚动切章，曾做过 Sheet 切章再改回；以参考视频为准，改交互后同步改文档，避免「代码一套文档一套」。

### Q29：你的优势？短板？

**优势示例：** 能闭环交付（方案→实现→打包→文档）；重视体验细节与可讲清取舍。  
**短板示例：** 本次未上登录与 PG，生产硬化还需一轮；若对方深挖原生性能优化，可坦诚「当前以产品还原优先，列表虚拟化等可后续加」。
